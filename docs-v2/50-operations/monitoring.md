## Structured logging (Pino)

The API uses Pino for structured JSON logging. Log level is configured via `LOG_LEVEL` env var (default: `info`).

In development: pretty-printed with `pino-pretty` transport.  
In production: JSON to stdout → Cloud Logging.

```typescript
// Usage in service (via request context)
const logger = getRequestContext().logger;
logger.info({ registrationId, eventId }, 'Registration confirmed');
logger.error({ error }, 'Badge generation failed');
```

**Never use `console.log` in service code.** Use the Pino logger from request context.

## Request context

Every request carries a `requestId` (auto-generated by Fastify, propagated via `x-request-id` header) through the entire async call chain via `AsyncLocalStorage`. This makes correlating log lines trivial:

```bash
# Find all logs for a specific request
gcloud logging read 'jsonPayload.requestId="req_abc123"' \
  --project=teranga-app-990a8 \
  --limit=50
```

## Error tracking (Sentry)

Sentry is initialized in `apps/api/src/server.ts`:

```typescript
initSentry({
  dsn: process.env.SENTRY_DSN,
  environment: process.env.APP_ENV,
  tracesSampleRate: 0.1,           // 10% of requests traced
  ignoreErrors: [                  // 4xx errors not reported
    /^(400|401|403|404|409|422|429)/
  ],
});
```

Sentry only captures **5xx errors** — client errors (4xx) are expected and not reported.

Enrich a Sentry capture with context:

```typescript
captureError(error, { requestId, userId, organizationId, resource: 'event' });
```

## Health probes

Two health endpoints for Cloud Run routing decisions:

```
GET /health   → always 200 while process is alive (liveness probe)
GET /ready    → 200 when Firestore + Auth are reachable (readiness probe)
```

Cloud Run is configured to:
- Stop routing to an instance if `/ready` returns non-200 (readiness failure)
- Restart an instance if `/health` returns non-200 three times in 30 seconds (liveness failure)

## Process-level error handling

```typescript
process.on('unhandledRejection', (reason) => {
  logger.error({ reason }, 'Unhandled promise rejection — logged, not crashing');
  captureError(reason);
  // never exit — unknown async rejection, likely recoverable
});

process.on('uncaughtException', (error) => {
  logger.fatal({ error }, 'Uncaught exception — initiating graceful shutdown');
  captureError(error);
  server.close(() => process.exit(1)); // drain in-flight requests first
});
```

## Graceful shutdown

Cloud Run sends `SIGTERM` with a 10-second grace period before `SIGKILL`. The API handles this:

```typescript
const shutdown = async () => {
  await server.close(); // drains in-flight requests
  process.exit(0);
};
process.on('SIGTERM', shutdown);
process.on('SIGINT', shutdown);
```

## Cloud Logging

Functions and Cloud Run both log to Cloud Logging automatically. Key queries:

```bash
# API errors (5xx)
gcloud logging read 'resource.type="cloud_run_revision" AND severity>=ERROR' \
  --project=teranga-app-990a8

# Cloud Functions errors
gcloud logging read 'resource.type="cloud_function" AND severity>=ERROR' \
  --project=teranga-app-990a8

# Slow requests (>2s)
gcloud logging read 'jsonPayload.responseTime>2000' \
  --project=teranga-app-990a8
```

## Firebase Performance Monitoring

The Flutter mobile app uses Firebase Performance Monitoring (imported in `pubspec.yaml`). Custom traces for:
- Event list load time
- Badge display time
- QR scan response time (offline vs online)

Web apps: Firebase Analytics is imported for the participant app. Analytics events are sent for key funnel steps (event view, registration start, registration complete).
